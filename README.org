#+TITLE: literate-agent
#+SUBTITLE: Shared literate-programming doctrine for AI agents

A Claude Code plugin and =CLAUDE.md=-importable rule aggregator.
Designed agent-neutral — Claude Code today, OpenCode / Codex /
Gemini ACP tomorrow.

* Why this exists
:PROPERTIES:
:CUSTOM_ID: why
:END:

Multiple repositories were duplicating the same
=.claude/rules/*.md= files and =CLAUDE.md= sections that codify
literate-programming discipline. Every doctrine update meant
editing N repos. This repo is the single source of truth.

Consumer repos import the doctrine with one line:

#+begin_src markdown
@~/projects/literate-agent/CLAUDE.md
#+end_src

…and Claude Code inlines the rules + language hints + cowork
protocols on every session.

* What's inside
:PROPERTIES:
:CUSTOM_ID: what
:END:

| Directory | Loaded by | Purpose |
|-----------+-----------+---------|
| =rules/= | =CLAUDE.md= =@=-import | Rule files — doctrine the agent should follow |
| =skills/= | Claude Code plugin | Auto-invoking skills (=docs-first=, =lp-research-audit=, =lp-section-shape=, =lp-dual-audience-check=, =lp-hint-elisp=, =lp-hint-python=, =lp-style-refactor=, =rebuild-lp-readme=) |
| =hooks/= | Claude Code plugin | =PreToolUse= / =PostToolUse= hooks (block edits to tangle outputs, auto-tangle on .org edit) |
| =commands/= | Claude Code plugin | Slash commands (=/lp-check=, =/lp-research-audit=, =/lp-section=) |
| =hints/= | lazy via skills | Per-language traps + cross-audience checklist |
| =templates/= | manual reference | Boilerplate (new module .org, new skill .org, README templates per consumer shape) |
| =scripts/= | =Makefile= / hooks | =build_index.py=, =build_readme.py=, =check_org_structure.py=, =measure-docs-first.sh= |
| =docs/= | manual reading | Background — LP philosophy, transfer-gradient research synthesis, config schema |
| =research/= | manual reading | Long-form research that grounded the doctrine |
| =literate-org/= | nested package | =literate-org= Python lib (tangling engine) — separate package, used by some scripts |

This repo is *not* a meta-repo: it has no =lp/= tree of its own,
no =repos/= submodules. Examples in =rules/= use placeholder
identifiers (=${PROJECT_NAMESPACE}-...=) so a consumer can adapt
without literal find/replace.

* Install
:PROPERTIES:
:CUSTOM_ID: install
:END:

Clone once at a stable path:

#+begin_src shell
git clone https://github.com/jingtaozf/literate-agent.git ~/projects/literate-agent
#+end_src

Then in any consumer repo's =CLAUDE.md=:

#+begin_src markdown
@~/projects/literate-agent/CLAUDE.md
#+end_src

For plugin-loaded skills + hooks + commands, launch Claude Code with:

#+begin_src shell
claude --plugin-dir ~/projects/literate-agent ...
#+end_src

Override the home path via =LITERATE_AGENT_HOME= if cloned
elsewhere.

* Per-consumer configuration — =.literate-agent/config.toml=
:PROPERTIES:
:CUSTOM_ID: config
:END:

Each consumer repo declares its project-specific values in its
own =.literate-agent/config.toml=. literate-agent scripts read
this file (discovered upward from CWD) to render placeholders
and choose the README shape.

Minimal example for a *plugin-consumer* repo (single repo using
the shared doctrine):

#+begin_src toml
[project]
namespace      = "code-agent"
display_name   = "Claude Agent SDK for Emacs"
author         = "Jingtao Xu"
repo_url       = "https://github.com/jingtaozf/emacs-agent"

[layout]
shape          = "plugin-consumer"  # vs "meta-repo" or "single-repo-lp"

[paths]
literate_agent_home = "~/projects/literate-agent"
#+end_src

For a *meta-repo* (many submodules under =repos/=):

#+begin_src toml
[project]
namespace      = "edo"
display_name   = "EDO Literate Meta-Repo"

[layout]
shape          = "meta-repo"

[paths]
lp_root        = "lp"
repos_root     = "repos"
#+end_src

Full schema: [[file:docs/config.org][docs/config.org]].

* Placeholder convention
:PROPERTIES:
:CUSTOM_ID: placeholders
:END:

Rule examples in =rules/*.md= use shell-style placeholders that
=scripts/build_readme.py= and =scripts/build_index.py= expand at
render time using the consumer's =config.toml=. Common ones:

| Placeholder | Expands to (example) |
|-------------+----------------------|
| =${PROJECT_NAMESPACE}= | =code-agent= (= symbol prefix in consumer code) |
| =${PROJECT_AUTHOR}= | =Jingtao Xu= |
| =${PROJECT_REPO}= | =https://github.com/jingtaozf/emacs-agent= |
| =${PROJECT_DISPLAY_NAME}= | =Claude Agent SDK for Emacs= |
| =${LP_ROOT}= | =lp= (meta-repo only) |
| =${REPOS_ROOT}= | =repos= (meta-repo only) |

When you write a new rule that contains a project-specific name
(e.g. a function symbol, a repo name), use the placeholder form,
not a literal. Readers who land on a literal will assume it's
the canonical name and copy it.

:CUSTOM_ID: layout
:END:

literate-agent/
├── CLAUDE.md              # @-importable aggregator
├── README.org             # this file

├── docs/                  # config schema, philosophy, research synthesis
├── research/              # long-form research grounding the doctrine
└── literate-org/          # nested Python package (tangling engine)
#+end_example
