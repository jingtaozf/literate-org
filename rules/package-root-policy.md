# Package Root Policy

> *Last-validated*: 2026-05-15
> *Review cadence*: quarterly — drop if 6 months without a triggering incident
> *Origin*: <meta-repo>; the principle generalises beyond that repo.

Adapted from three submodule variants:
`repos/<wisdom-store>/.claude/rules/package-root-policy.md`,
`repos/<python-foundation>/.claude/rules/package-root-policy.md`,
`repos/<scout-server>/.claude/rules/package-root-policy.md`.
Merged into one rule with submodule-scoped sections because each
submodule has a different package root and a different list of
pre-approved top-level entries.

## Common rationale

A flat package root rots fast as the project grows. Every consumer
runs `from <pkg> import *` somewhere, and surprise top-level names
break tab-completion and linters. Subpackages make the public
surface obvious and let `__init__.py` stay tiny.

The unifying rule across all three submodules: **do not place new
modules directly at the package root unless they are on the
submodule's pre-approved list below.** All other modules belong in a
sub-package.

## Scope-specific lists

### `repos/<wisdom-store>/` — `src/pcr_skill_networking/`

Pre-approved root entries:

- Global configuration (`config.py`)
- Domain models shared across all modules (`domain.py`)
- Package-level CLI entry points (`cli.py`)

Sub-packages:

- `utils/` — shared infrastructure (LLM, embedding, metrics, tracing, usage tracking)
- `server/` — FastAPI server
- `client/` — client protocol + remote client
- `router/` — wisdom router orchestrator
- `<storage>/` — storage layer
- Stage packages: `parser/`, `validator/`, `reasoner/`, `retriever/`, `curator/`

### `repos/<python-foundation>/` — `knowledge_utils/`

This package root is the **public surface 5+ downstream repos depend on**;
every new top-level entry becomes part of the implicit API contract.

| Path | Role |
|------|------|
| `__init__.py` | Public re-exports (caching, tracing, observability, configure) |
| `_version.py` | Auto-generated by `setuptools_scm`; do not hand-edit |
| `_config.py` | Runtime config singleton (`WGUConfig`, `configure`, `current`, `getenv`) |
| `caching/` | `SqliteObjectCache` and friends (5+ consumer repos depend on this) |
| `tracing/` | `register_tracing` (OpenAI auto-instrumentation) |
| `observability/` | `setup_file_tracer_provider`, `setup_oauth2_otlp_instrumentation`, parsers |
| `llm/` | `LLMClient` baseline (since v0.2.0) |
| `utils/` | Shared pipeline utilities (since v0.2.0) |
| `api/` | FastAPI helpers — `BearerAuth` for dual JWT + API-key auth (opt-in via `[api]` extra). Consumers: <scout-server>, <app>-server, <enhance-server>. |
| `db/` | Postgres helpers — `advisory_lock` context manager + `AdvisoryLockHeldError` (opt-in via `[db]` extra). Consumers: <asset-management> (cycle lock); <scout-server> / <wisdom-store> are likely follow-on consumers for ingest / write-side locks. |
| `examples/` | Documentation-grade usage examples |

Any PR that introduces a new `knowledge_utils/<thing>/` must:

1. Justify why the role isn't covered by an existing subpackage in the
   PR description.
2. Demonstrate at least *2 consumer repos* will use it (consistent with
   the "common = used by 2+ repos" filter applied at v0.2.0).
3. Not auto-import from `knowledge_utils/__init__.py` if it pulls
   heavy deps (litellm / opentelemetry / numpy / boto3 / anthropic /
   google-genai). Heavy deps go behind explicit submodule imports.

Adding a flat-file under the package root is discouraged in nearly
all cases. Acceptable carve-outs: pure runtime-config / wiring (the
existing `_config.py` is the model), or a migration shim that will
be removed within one minor version (must be flagged in `CHANGELOG.md`
as deprecated on day one).

### `repos/<scout-server>/` — `skill_scout/`

Pre-approved root entries:

- Global configuration (`config.py`)
- Domain models shared across all modules (`domain.py`)
- Cross-module interfaces (`protocols.py`)
- Canonical package hashing (`packaging.py`)
- Package-level CLI entry point (`cli.py`)
- The dependency-injection factory (`factory.py`)

Sub-packages:

- `vendor/mega_code/` — vendored LLM + utils code (treat as read-only)
- `obs/` — OpenTelemetry / step-logger wiring
- `cache/` — cache backend wiring (thin wrapper around vendored)
- `llm/` — `Chat` and `Embedder` implementations over vendored LLMClient
- `store/` — persistence (SQLite today, Postgres later)
- `repositories/` — `SkillRepository` implementations (git branch, skills.sh, SkillsMP)
- `pipeline/` — `PipelineStep` classes (one per step)
- `gates/` — `Gate` implementations (Gate1, Gate2, Gate3)
- `reporter/` — `Reporter` implementations
- `prompts/` — YAML prompt files (no Python)

## Enforcement

- All three submodules: reviewed during PR review.
- `<python-foundation>` only: the architecture test
  `tests/unit/test_architecture.py::test_package_root_only_has_approved_entries`
  asserts the top-level entries match the table above; new entries
  must update both the table and the test in the same PR.
